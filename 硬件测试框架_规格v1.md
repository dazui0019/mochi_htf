# 硬件测试框架规格 v1

## 1. 目标与范围
- 运行平台：Windows 工控机。
- 运行形态：Web UI，支持局域网多端访问。
- 使用方式：单用户、单任务执行（同一时刻只允许一个测试任务）。
- 框架定位：仅提供测试编排与执行框架，不直接实现底层设备驱动能力。
- 扩展方式：通过插件机制扩展设备操作能力。

## 2. 核心能力
- 用例管理：JSON 用例加载、查看、编辑、覆盖保存。
- 用例生命周期：支持创建与删除。
- 层级模型：`Test Item -> Test Step`。
- 工步类型：
  - `Action`：只执行，不做表达式判定。
  - `Verify`：执行后使用表达式判定 `Pass/Fail`。
- 执行结果状态：
  - `Pass`：执行成功且（若为 Verify）判定成立。
  - `Fail`：执行成功但 Verify 判定不成立，继续后续步骤。
  - `Error`：插件异常或超时，熔断当前 Item 剩余步骤，继续下一个 Item。
- 执行控制：开始、停止（协作式停止，当前 Step 结束后停止后续步骤）。
- 历史记录：保留最近 1000 次执行。
- 报告输出：历史详情页查看、JSON 导出、Excel 导出。

## 3. 技术方案
- 后端：FastAPI
- 前端：Vue 3（CDN 版，便于单机部署）
- 存储：SQLite
- 数据目录约定：
  - `./testcases`：用例文件
  - `./plugins`：插件文件
  - `./reports`：导出报告
  - `./data/app.db`：历史记录数据库

## 4. 用例模型

### 4.1 TestCase
- `id: string`
- `name: string`
- `version: string`
- `items: TestItem[]`

### 4.2 TestItem
- `id: string`
- `name: string`
- `steps: TestStep[]`

### 4.3 TestStep
- `id: string`
- `name: string`
- `type: Action | Verify`
- `plugin: string`
- `action: string`
- `params: object`
- `expr?: string`（Verify 必填）
- `timeout?: number`（秒，默认 5）

### 4.4 示例
```json
{
  "id": "case_power_on_001",
  "name": "上电基础测试",
  "version": "1.0.0",
  "items": [
    {
      "id": "item_boot",
      "name": "启动检查",
      "steps": [
        {
          "id": "step_send_cmd",
          "name": "发送启动指令",
          "type": "Action",
          "plugin": "sim_device",
          "action": "send_command",
          "params": {
            "cmd": "BOOT"
          },
          "timeout": 5
        },
        {
          "id": "step_read_state",
          "name": "读取状态码",
          "type": "Verify",
          "plugin": "sim_device",
          "action": "read_state",
          "params": {},
          "expr": "result == 1",
          "timeout": 5
        }
      ]
    }
  ]
}
```

## 5. 插件规范
- 插件目录：`./plugins`。
- 发现方式：启动或刷新时扫描目录下 `*.py` 文件。
- 插件入口（满足其一）：
  - `create_plugin() -> object`
  - `class Plugin`（可实例化）
- 插件实例要求：
  - `name: str`（推荐）
  - `run(action: str, params: dict) -> Any`
  - `actions() -> list[str]`（可选，用于 UI 展示）
- 运行时上下文注入：
  - 执行器会在每个 Step 执行前向 `params` 注入 `__htf_context`（框架保留字段）。
  - 字段包含：`run_id`、`case_id`、`item_id`、`step_id`、`step_name`、`reports_dir`。
  - 插件应忽略未知字段；业务参数不应使用 `__htf_` 前缀。
- 附件结果约定（推荐）：
  - 插件可在 `result.artifacts`（数组）返回附件元数据。
  - 每个附件建议包含：`type`、`label`、`path`、`url`。
  - 推荐类型：`image`、`log`、`csv`、`waveform`。
  - `waveform` 类型可选包含 `series` 数组（如电压/电流通道及采样点）。
  - 典型格式：`{"artifacts":[{"type":"image","path":"reports/<run_id>/artifacts/xxx.png","url":"/api/reports/<run_id>/artifacts/xxx.png"}]}`。

## 6. 表达式引擎
- Verify 表达式仅允许访问变量：`result`。
- 允许基础布尔、比较、算术表达式。
- 禁止函数调用、属性访问、导入、执行语句。
- 表达式必须返回布尔值。

## 7. 执行规则
- 执行顺序：Item 串行，Step 串行。
- Fail 不熔断；Error 熔断当前 Item。
- 默认超时：5 秒（可按 Step 覆盖）。
- 停止语义：收到停止请求后，不再开始新 Step，等待当前 Step 结束。
- 运行最终状态：
  - `Pass`：无 Fail/Error。
  - `Fail`：存在 Fail 且无 Error。
  - `Error`：存在 Error 或运行异常。
  - `Stopped`：人工停止。

## 8. 报告与历史
- 报告 JSON 包含：运行基本信息、Item/Step 详细结果、统计摘要。
- Step 结果可包含附件元数据（如示波器截图），附件文件存放于 `reports/<run_id>/artifacts/`。
- Excel 每行对应一个 Step。
- Excel 最小列：`Item名`、`Step名`、`类型`、`结果`、`错误信息`、`附件`。
- 历史保留策略：最近 1000 次，超出自动淘汰最旧记录。
- 历史入口：主页展示历史摘要；点击测试用例名称进入详情页。
- 导出入口：仅在历史详情页提供 JSON/Excel 导出按钮。

## 9. API（MVP）
- `GET /api/plugins`
- `GET /api/cases`
- `GET /api/cases/{case_id}`
- `PUT /api/cases/{case_id}`
- `DELETE /api/cases/{case_id}`
- `POST /api/runs/start`
- `POST /api/runs/stop`
- `GET /api/runs/current`
- `GET /api/runs/history`
- `GET /api/reports/{run_id}.json`
- `GET /api/reports/{run_id}/artifacts/{artifact_path}`
- `GET /api/reports/{run_id}.xlsx`

## 10. UI 规则（当前实现）
- 用例加载后默认只读。
- 新建用例后自动加载并进入编辑模式。
- 进入编辑模式后可编辑；保存后自动退出编辑模式。
- 仅在只读模式允许开始测试。
- 只读模式隐藏 Item/Step 的新增、删除、上下移操作按钮。
- Item 与 Step 均为可折叠卡片。
- Item 与 Step 默认折叠；保存后保留当前折叠状态。
- 新增入口采用卡片式 `+` 交互。
- Step 顶栏直接编辑 `type` 与 `name`，并在右侧显示实时状态。
- Step 内容区编辑 `plugin/action/timeout/params/expr`，其中 `plugin/action` 为下拉选择。
- 历史记录列表不显示操作列；测试用例名为超链接，点击进入详情页。

## 11. 验收标准
- 可自动发现并调用模拟插件。
- 可加载、编辑、保存 JSON 用例。
- 执行中正确区分 Pass/Fail/Error。
- Error 仅熔断当前 Item，继续后续 Item。
- 停止按钮按“当前 Step 结束后停止”执行。
- 可查看历史详情并导出 JSON/Excel。
